## 二、控制运营成本的技巧

### 实际采用的

| # | 技巧 | 具体做法 | 效果 |
|---|------|---------|------|
| 1 | 云端 STT 替代本地模型 | 使用 DashScope Paraformer-v2 API，放弃本地 faster-whisper | 识别率 >95%，延迟从 8s 降至 1-2s，按量计费成本可控 |
| 2 | 动态帧数控制 | PTT 窗口内均匀采样 3-5 帧，不发多余画面 | 每次 LLM 调用图片 token 控制在 5000 以内 |
| 3 | 视觉路由器 | DeepSeek-chat 判断问题是否需要画面，非视觉问题 0 帧 | 非视觉对话 LLM 耗时减半，token 费用相应减少 |
| 4 | 单模型策略 | 只使用 qwen3.7-plus，不维护多模型路由 | 减少额外的模型切换调用和代码维护成本 |
| 5 | PTT 窗口隔离 | 只发送说话期间的帧，空闲画面不发给 LLM | 避免无关画面消耗 token |
| 6 | 帧分辨率降低 | 512px → 384px，JPEG 质量 0.6 → 0.55 | 图片体积减少约 40%，相应减少 token |
| 7 | 流式输出 | LLM 边生成边返回，首 token 到达即显示 | 用户感知等待时间明显缩短 |

### 考虑但未采用

| # | 技巧 | 不采用原因 |
|---|------|-----------|
| 1 | 本地 STT（faster-whisper） | tiny 模型中文识别率低，small 模型 CPU 推理太慢（8-9s），云端更快更准 |
| 2 | pHash 感知哈希去重 | 对小物体（手机、杯子）不敏感，阈值调优困难，增加复杂度 |
| 3 | 帧动作检测（frame diff） | 低分辨率下物品变化太小，误判率高，最终去掉 |
| 4 | TTS 语音合成 | MVP 阶段文字输出已满足需求 |
| 5 | 多模型适配器抽象层 | 单一场景不需要，已移除 |
| 6 | 图片差异化压缩（人脸/文字/物体自适应质量） | 实现复杂度高，当前固定压缩已满足需求 |
| 7 | 端侧轻量视觉模型预处理（如 CLIP 截图再发给大模型） | 增加端侧推理负担，ROI 不足 |
