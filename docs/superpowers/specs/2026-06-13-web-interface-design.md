## 需求概述

开发一款浏览器端 AI 视觉对话应用。用户打开网页即可使用摄像头和麦克风与 AI 对话，AI 能看到画面、听懂语音并给予恰当回应。

原始要求：
- 打开摄像头与麦克风，让 AI 看到视频内容、听到用户说话
- 综合考虑视觉内容理解准确性、语音交互自然度与流畅性
- 端云协同的成本控制策略
- 提交设计文档，包含用户故事和成本控制技巧

## 架构设计

```
浏览器 (HTML/JS)
  ├─ getUserMedia → 摄像头 + 麦克风
  ├─ PTT 按钮 → 录音 + 抓帧
  ├─ WebSocket → JSON/二进制消息
  └─ 对话气泡渲染

FastAPI 后端 (server.py)
  ├─ GET / → 静态页面
  ├─ WS /ws → WebSocket 端点
  ├─ STT → faster-whisper (本地)
  └─ LLM → 千问 DashScope API
```

### 通信协议

```
客户端 → 服务器:
  {"type": "ptt_start"}
  {"type": "ptt_stop", "frames": ["base64..."]}
  {"type": "audio", "data": "<WAV base64>"}

服务器 → 客户端:
  {"type": "status", "text": "转写中..."}
  {"type": "transcript", "text": "你好"}
  {"type": "response", "text": "你好！"}
  {"type": "error", "text": "未识别到语音"}
```

### 音频格式

- 浏览器端：`AudioContext` + `ScriptProcessorNode` 采集 PCM，编码为 WAV（44 字节头 + int16 PCM），base64 传输
- 服务器端：Python `wave` 标准库解码 → float32 → STT

### 帧抓取策略

| 状态 | 抓取间隔 | 说明 |
|------|:--:|------|
| 空闲 | 3s | 后台低频，维持帧缓冲 |
| PTT 期间 | 0.5s | 高频抓取 |
| 发送 LLM | 3-10 帧 | 按录音时长动态分配 |

| 录音时长 | 发 LLM 帧数 |
|:--:|:--:|
| ≤3s | 3 |
| 3-10s | 5 |
| 10-20s | 8 |
| 20-60s | 10 |

## 成本控制策略

### 采用的技巧

| 策略 | 实现 | 效果 |
|------|------|------|
| STT 本地推理 | faster-whisper tiny 模型 (~70MB)，CPU int8 量化 | 零 API 费用，延迟 ~500ms |
| 帧数量上限 | 按录音时长动态 3-10 帧，绝对上限 10 | 每次 LLM 调用图片 token ~5000-10000 |
| 空闲低频抓帧 | 不对话时 3s 一帧，对话时 0.5s | 减少无效帧，降低带宽和 token |
| 单模型 | 只用 qwen3.7-plus，无需多模型路由 | 简化调用链，避免冗余开销 |
| 本地传输 | 帧和音频走 localhost WebSocket | 零带宽成本 |

### 考虑但未采用的

| 策略 | 原因 |
|------|------|
| VAD 替代 PTT | PTT 按钮更可靠，避免误触发浪费 LLM 调用 |
| 云端 STT | 本地 faster-whisper 已经够快够准 |
| 对话历史摘要 | 当前 max 10 轮开销不大，暂不需要 |
| 自动停止录音 | 静音检测可能误判，PTT 按钮更直观 |

## 错误处理

| 情况 | 前端行为 |
|------|---------|
| 录音 < 0.3s | 提示"录音太短"，不发送 |
| STT 返回空 | 显示"未识别到语音，请重试" |
| LLM API 异常 | 显示"AI 暂时不可用，稍后重试" |
| 权限被拒绝 | 页面提示"需要摄像头和麦克风权限" |
| WebSocket 断开 | 显示"连接断开"，自动重连 |
