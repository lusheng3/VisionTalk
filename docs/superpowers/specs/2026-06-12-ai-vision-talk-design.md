## 1. 项目概述

**AI 视觉对话助手**是一款基于 Web 的多模态 AI 对话应用。用户通过浏览器打开摄像头和麦克风，AI 能够看到摄像头画面、听到用户语音，并以语音+文字的方式给予回应。核心体验类似 ChatGPT 高级语音模式 + 视觉：AI 作为对话伙伴，同时感知视觉和听觉信息，自然交流。

### 核心体验特征

| 特征 | 说明 |
|------|------|
| 👁️ 持续视觉感知 | AI 能看到摄像头画面，感知环境、物体、人物 |
| 🗣️ 语音自然对话 | 像和朋友聊天一样说话，AI 语音回复 |
| 🧠 多模态理解 | AI 融合视觉+语音信息，指着画面问"这是什么"无需描述 |
| 💬 记忆上下文 | 记住之前说过的话和看过的画面，支持多轮深入对话 |

### 2.2 组件职责

| 组件 | 位置 | 职责 | 关键技术 |
|------|------|------|----------|
| Camera/Mic Capture | 前端 | 获取摄像头视频流和麦克风音频流 | MediaDevices.getUserMedia() |
| WebSocket Client | 前端 | 双向通信，发送截图帧和音频，接收 AI 响应 | 原生 WebSocket API |
| Audio Playback | 前端 | 流式播放 TTS 音频 | AudioContext.decodeAudioData() |
| WebSocket Server | 后端 | 接收音视频数据，分发到各处理模块 | FastAPI + websockets |
| VAD | 后端 | 检测用户是否在说话，过滤静音段 | Silero VAD (ONNX) |
| Frame Extractor | 后端 | 说话时截取关键帧，相邻帧去重，分辨率压缩 | OpenCV + SSIM |
| STT | 后端 | 语音转文字 | Whisper (本地 tiny/base) |
| Orchestrator | 后端 | 组装多模态 prompt，调用 LLM API，成本控制 | 适配器模式 |
| TTS | 后端 | 文字转语音合成 | Edge TTS (免费) |
| Session Manager | 后端 | 管理对话历史和上下文 | 内存 + 可选文件持久化 |

### 2.3 视觉采集策略：语音触发截帧（方案 C）

**核心原则**：不持续传输视频帧。摄像头仅在本地预览，只有当用户说话时才截取关键帧发送给 AI。

**三种触发时机**：

1. **语音触发（主）**：用户开始说话、说话中间、结束各截 1 帧，共 3 帧
2. **画面变化触发（辅）**：SSIM 对比检测到画面剧烈变化时，捕获变化前后各一帧
3. 截帧后处理：相邻去重 → 分辨率缩放到 512-768px → JPEG 编码 → base64

**与持续视频流的成本对比**：

| | 持续视频流 | 三帧截图 (方案 C) |
|---|---|---|
| 每分钟图片数 | 30-60 张 | 3-8 张 |
| 每分钟 image tokens | ~5000-10000 | ~300-800 |
| API 成本/分钟 | ¥0.2-0.5 | ¥0.01-0.04 |
| 节省比例 | - | **10-20x** |

## 4. 数据流

### 4.1 端到端对话流程

```
1. 用户打开页面 → 浏览器请求摄像头+麦克风权限
2. 摄像头预览显示在页面上（仅本地，不传输）
3. 用户说话（PTT 按住 或 持续模式 VAD 检测）
   ├── 后端 VAD 检测语音段开始 → 触发截帧
   │   ├── 截帧1：语音开始时的画面
   │   ├── 截帧2：语音中间的画面
   │   └── 截帧3：语音结束时的画面
   └── 后端 STT (Whisper) 转写语音为文字
4. Orchestrator 组装多模态 Context：
   {
     "system": "你是视觉对话助手，能看见用户和周围环境...",
     "history": [前3轮对话],
     "images": [截帧1, 截帧2, 截帧3],  // base64
     "user_text": "这是什么植物？"
   }
5. 调用云端多模态 LLM → 获取文字回复
6. TTS 合成语音 → WebSocket 推送音频块 + 文字 → 前端播放
7. 对话记录存入 Session Manager → 回到 IDLE 等待下一轮
```

### 4.2 画面变化触发流程（补充）

```
摄像头帧持续监控（仅后端分析，不发送）
  → 每 2 秒计算当前帧与上一截图帧的 SSIM
  → 若 SSIM < 0.7 → 判定为画面剧变
  → 捕获变化前后各一帧，标记为"场景切换"
  → 附加到下一轮对话的 Context 中
```

## 6. 端云协同 & 运营成本控制

### 6.1 核心原则

**端侧做减法，云端做理解**。所有能在本地完成的计算（语音检测、图像去重、压缩、STT、TTS）都在本地完成，仅将最精简、最有信息量的数据发送给云端 AI。

### 6.2 成本控制技巧清单

| # | 技巧 | 说明 | 预估节省 | 想到 | 实际采用 |
|---|------|------|----------|------|----------|
| 1 | **帧去重 (SSIM)** | 相邻帧 SSIM > 0.95 视为相同，只保留一张 | 减少 70-90% 图片 | ✅ | ⬜ |
| 2 | **VAD 静音过滤** | 切掉前后静音和呼吸声，只处理有效语音 | 减少 30-50% 音频 | ✅ | ⬜ |
| 3 | **分辨率压缩** | 图片缩放到 ≤768px，降低 image token 消耗 | token 从 ~1000 → ~150 | ✅ | ⬜ |
| 4 | **Whisper 本地 STT** | 语音识别全在本地跑，不走付费 API | 零 STT API 成本 | ✅ | ⬜ |
| 5 | **Edge TTS 免费合成** | 语音合成用微软 Edge TTS（免费），不调付费 API | 零 TTS API 成本 | ✅ | ⬜ |
| 6 | **模型分级调用** | 简单问答用廉价模型 (Qwen-VL-Plus)，复杂分析用高价模型 (Qwen-VL-Max) | 简单问答降 80%+ | ✅ | ⬜ |
| 7 | **Context 窗口管理** | 仅保留最近 N 轮完整对话，超出部分做摘要压缩 | 控制每轮 prompt token | ✅ | ⬜ |
| 8 | **API 调用去抖** | 用户连续说话时去抖 (debounce 1.5s)，避免重复调用 | 减少 30%+ 调用次数 | ✅ | ⬜ |
| 9 | ~~响应缓存~~ | ~~高频相似问题缓存通用回复~~ | - | ✅ | ❌ 暂不采用 |
| 10 | ~~流式传输~~ | ~~LLM 流式返回 → TTS 边生成边播放~~ | - | ✅ | ⚠️ P2 理想功能 |

### 6.3 成本估算

假设用户每天使用 30 分钟，每分钟约 2 轮对话：

| 项目 | 单轮消耗 | 日消耗 (60轮) | 月消耗 (1800轮) |
|------|----------|--------------|----------------|
| Image tokens | ~300 | ~18,000 | ~540,000 |
| Text tokens (in) | ~200 | ~12,000 | ~360,000 |
| Text tokens (out) | ~300 | ~18,000 | ~540,000 |
| **月 API 成本 (Qwen-VL-Max)** | - | - | **≈ ¥12-18** |
| **月 API 成本 (Qwen-VL-Plus)** | - | - | **≈ ¥2-4** |
| **月 API 成本 (GLM-4V)** | - | - | **≈ ¥0.1-0.3** |

> 注：STT (Whisper 本地) 和 TTS (Edge TTS) 为零成本。成本仅来自云端多模态 LLM 调用。

## 7. 待定/风险项

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
